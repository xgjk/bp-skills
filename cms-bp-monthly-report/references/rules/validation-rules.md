# 报告合规性校验与清洗规则

> **MANDATORY RULES — 最高优先级**
> 校验未全部通过时，严禁调用保存接口。

---

## 一、校验执行规范

### 执行时机

`render_full_report` 生成 `report_selfcheck.md` 之后、`save_openclaw_report` 之前。

### 执行方式

1. 读取 `report_selfcheck.md` 完整内容
2. 逐条对照下方校验清单，每条输出 `✅ 通过` 或 `❌ 未通过：具体原因`
3. 全部通过 → 保存；任何一项未通过 → 按回退规则修正

### 脚本已保证的项（AI 无需重复校验）

以下由 `render_full_report` 脚本硬编码保证，AI 可以跳过：
- 报告章节完整性、章节顺序（脚本固定拼接）
- 1.2 灯色分布格式（脚本硬编码 `text` 代码块）
- 2.1 总览表 7 列格式、2.2 目标明细 4 子章节结构
- KR 6 子字段、举措 5 子字段（脚本逐字段渲染）
- 编码完整性（脚本直接从 JSON 读取 `fullLevelNumber`）
- 四灯判断块结构和行数（`_render_lamp_block` 硬编码）
- 证据引用格式（脚本从 JSON 原样渲染，格式由 Skill A 保证）
- 第 3/4 章链接内容（脚本写死固定链接）
- 灯色一致性和分布统计（脚本从同一字段读取渲染到所有位置）
- 异常灯人工确认区（`_render_lamp_block` 自动处理）

## 二、AI 校验清单（4 项）

| 序号 | 校验项 | 校验方法 |
|------|--------|---------|
| 1 | **1.1 结论格式** | 必须包含两个独立行：`一句话优势：...` 和 `一句话短板：...`，不可合并为一段话 |
| 2 | **1.3 偏差点格式** | 若无偏差：必须输出"本月无重大偏差点。"。若有偏差（最多3条）：每条必须有 `偏差点`/`影响`/`原因假设`/`下月纠偏方向` 4个子字段 |
| 3 | **附录 A.2 条数一致** | 正文中出现的 R 编号总数必须等于附录证据索引表中的条数 |
| 4 | **数据完整性** | 各目标章节中不应出现空白字段（如承诺口径、本月实际为空），若原始 JSON 数据为空应有自然语句填充 |

## 三、语言清洗规则（5 条）

### 规则 1：禁止技术字段泄漏

正文中不得出现 API 原始字段名，包括但不限于：
`reportId`、`authorEmpId`、`createTime`、`taskId`、`groupId`、`employeeId`、`type=manual`、`type=ai`、`contentHtml`、`planDateRange`、`statusDesc`、`measureStandard`、`fullLevelNumber`、`upwardTaskList`、`reportTaskMapping`。

### 规则 2：句式自然化

以下表述**严禁出现**：
- ~~"XX 为空"~~ → "本月尚未收到相关汇报"
- ~~"数据不完整"~~ → "当前可获取的信息有限，建议补充"
- ~~"举证原数据不完整"~~ → "本月关联的工作汇报内容较少，尚不足以全面评估"
- ~~"本月最关键的进展为空"~~ → "本月暂无可明确标注的关键进展，建议关注 XX 方向"
- ~~"无关联汇报"~~ → "本月该事项下暂未收到工作汇报"

### 规则 3：禁止空值直出

空字段必须改写为有引导意义的自然语句。

### 规则 4：禁止模板括号注释泄漏

**任何以中文括号 `（）` 包裹的、用于指导 AI 生成行为的说明文字**，一律不得出现在最终报告中。

### 规则 5：禁止系统流程说明泄漏

以下内容不得出现在最终报告中：
- 任何提及 Phase 1a/2a/3b 等内部流程步骤的文字
- "字段与台账一致"、"页内锚点"等内部实现说明
- "约束："、"AI 指引："、"注意："开头的系统提示
- 任何 `<!-- ... -->` HTML 注释标签及其内容

## 四、校验失败回退规则

1. **全局结论问题**（1.1/1.3 内容不当）：修正 `conclusion_data.json`，重新执行 `render_full_report`，然后重新校验
2. **格式/灯色问题**：重新执行 `render_full_report` 即可
3. 同一问题最多重试 **2 次**，仍不通过则 `update_report_status --status 2`，终止流程
