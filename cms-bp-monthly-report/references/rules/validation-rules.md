# 报告合规性校验与清洗规则

> **MANDATORY RULES**
> 本文件中的所有规则为强制约束。AI 在 Step 3d（报告拼接与合规性校验）时，必须逐条执行全部校验项和清洗规则，不得跳过、简化或自行推断替代方案。
> 校验未全部通过时，严禁进入保存草稿流程。违反任何一条规则将导致报告验证失败。

本文件定义了 BP 月报自查报告在发送前必须通过的全部校验项、失败处理机制、语言清洗规则和附录搬运规则。

---

## 一、合规性校验清单

对写入的报告文件逐项校验以下清单，**全部通过后方可进入保存草稿流程**。

| 序号 | 校验项             | 校验标准                                                                                              |
|------|-----------------|---------------------------------------------------------------------------------------------------|
| 1 | **灯判断块结构**      | 黄灯、红灯、黑灯判断块，必须严格按照 report-template-bp-self-check.md 四灯判断块标准模板结构输出                                 |
| 2 | **1.1 报告格式完整**  | 严格对照 report-template-bp-self-check.md，检查生成的报告，是否符合模版                                              |
| 3 | **1.2 灯色分布格式**  | 必须使用 ` ```text ` 代码块，分两部分：参与自查目标数 + 四色灯统计（🟢/🟡/🔴/⚫各一行）+ 未参与自查的 ★ 未启动统计，共 7 行                    |
| 4 | **1.3 偏差点子字段**  | 每条偏差必须有 `偏差点` / `影响` / `原因假设` / `下月纠偏方向` 4 个子字段                                                   |
| 5 | **2.1 总览表列数**   | 必须 7 列：目标编号 / BP目标 / 本月承诺口径 / 本月实际 / 证据引用 / 目标灯色 / 结论一句话。所有目标均列入，被排除目标用 ★ 标记、结论写"未启动"、证据引用留空      |
| 6 | **目标明细 4 子章节**  | 每个参与自查的目标必须包含：承诺与实际对照 → 关键成果达成与举措推进 → 偏差问题与原因分析 → 目标级综合灯色结论                                       |
| 7 | **KR 级完整分析单元**  | 每个参与自查的 KR 必须有 6 个子字段：衡量标准 / 本月结果 / 距离衡量标准 / 环比上月 / 证据 / 判断理由（KR 不嵌入四灯判断块）                        |
| 8 | **举措级层级结构**     | 每个举措必须有 5 个子字段：推进动作摘要 / 对结果支撑【强/中/弱】 / 当前进度（含量化）/ 证据 / 举措级四灯判断块                                   |
| 9 | **证据引用仅编号**     | 正文中 R/RP 引用格式：`[R编号](huibao://...)`                                                               |
| 10 | **四灯判断块行数**     | 绿灯 = 2 行；黄灯/红灯/黑灯 = 8 行（指 8 个 `<span>` 内容行；`<div>` 和 `</div>` 包裹标签照常输出但不计入行数）。仅适用于举措级和目标级，KR 级无四灯判断块 |
| 11 | **第 3 章仅含链接**   | Section 3 严格按模板：仅输出一个年度结果预判评分链接，不得包含自我定性、结论解释或其他文字                                                |
| 12 | **附录 A.2 条数一致** | R 编号总数必须等于证据台账中的条数，不可多也不可少                                                                        |
| 13 | **语言清洗 5 条规则**  | 无技术字段泄漏、无空值直出、无模板括号注释、无系统流程说明、无 HTML 注释（详见本文第三章）                                                  |
| 14 | **数据完整性**       | 各目标章节中的所有 R 编号、灯色判断、偏差字段是否完整搬入正文，不可遗漏                                                             |
| 15 | **编码完整性**       | 各个目标、成果、举措编码完整、正确                                                                                 |
| 16 | **第 4 章仅含链接**   | Section 4 严格按模板：仅输出一个月度汇报入口链接，不得包含自我定性、结论解释或其他文字                                                  |

## 二、校验失败回退规则

1. 以**目标**为粒度定位失败项，仅回退修正该目标对应的章节，无需重新生成整篇报告；
2. 修正后**仅对该目标章节**重新执行合规性校验；
3. 同一目标最多重试 **2 次**，若仍不通过则调用 `update_report_status` 将状态标记为失败（status=2），并在 fail_reason 中写明具体不通过的校验项编号和内容，终止流程。

## 三、语言清洗规则

报告组装完成后、写入文件前必须执行。对全文逐段扫描，确认以下五条规则全部通过后才能写入文件：

### 规则 1：禁止技术字段泄漏

正文中不得出现 API 原始字段名，包括但不限于：
`reportId`、`authorEmpId`、`createTime`、`taskId`、`groupId`、`employeeId`、`type=manual`、`type=ai`、`contentHtml`、`planDateRange`、`statusDesc`、`measureStandard`、`fullLevelNumber`、`upwardTaskList`、`reportTaskMapping`。

如需表达相关含义，必须改用自然语言。

### 规则 2：句式自然化

所有描述采用"主语 + 谓语 + 宾语"的自然句式。以下表述**严禁出现**：
- ~~"XX 为空"~~ → "本月尚未收到相关汇报"
- ~~"数据不完整"~~ → "当前可获取的信息有限，建议补充"
- ~~"举证原数据不完整"~~ → "本月关联的工作汇报内容较少，尚不足以全面评估"
- ~~"本月最关键的进展为空"~~ → "本月暂无可明确标注的关键进展，建议关注 XX 方向"
- ~~"无关联汇报"~~ → "本月该事项下暂未收到工作汇报"

### 规则 3：禁止空值直出

空字段必须改写为有引导意义的自然语句。

### 规则 4：禁止模板括号注释泄漏

最终报告是直接发送给员工阅读的，章节标题中的括号说明文字一律不得出现在最终报告中。以下为必须清洗的映射表：
- ~~`（目标主线）`~~ → 标题中不加此后缀
- ~~`（先给结论，再给依据）`~~ → 删除
- ~~`（四档）`~~ → 删除
- ~~`（目标级）`~~ → 删除
- ~~`（可选，最多 3 条）`~~ → 删除
- ~~`（本报告主体，按目标动态生成）`~~ → 删除
- ~~`（承诺 vs 实际，表格）`~~ → 删除
- ~~`（逐目标展开：承诺对照 + 结果 + 举措 + 偏差问题）`~~ → 删除
- ~~`（可验收口径）`~~ → 删除
- ~~`（按成果层级展开）`~~ → 删除
- ~~`（若无偏差写"本目标本期无重大偏差"）`~~ → 删除
- ~~`（必填）`~~ → 删除
- ~~`（必须输出）`~~ → 删除
- ~~`（两句以内）`~~ → 删除
- 通用规则：**任何以中文括号 `（）` 包裹的、用于指导 AI 生成行为的说明文字**，一律不输出

### 规则 5：禁止系统流程说明泄漏

以下内容不得出现在最终报告中：
- "以下自 Step 3b 证据台账原样搬入"或类似提及内部步骤编号（Step 3a/3b/3c/3d 等）的文字
- "字段与台账一致；R 列已加页内锚点"等内部实现说明（含任何提及"页内锚点"的文字）
- "约束：本章小节数量必须随…"等以"约束："开头的系统提示
- "AI 指引："开头的任何文字
- 任何 `<!-- ... -->` HTML 注释标签及其内容

## 四、附录搬运规则

- 报告的证据索引附录由 `assemble_report` 脚本从 `evidence_ledger.md` **直接读取并原样搬入**
- 附录 A.2 中的 R 编号条数必须与证据台账中的条数完全一致，不得遗漏或增加
- 附录 A.3 上月参考信息由 `build_evidence_ledger` 从 `prev_month.json` 自动生成 RP 编号索引
- 若台账文件不存在或为空，必须回退重新执行对应步骤，**不得跳过或伪造**
