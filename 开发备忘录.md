# 开发备忘录

> 目标：记录开发过程中遇到的问题与解决方案，后续遇到类似问题优先在此查找。

## 2026-04-08

### 12) 组织月报模板需要同时覆盖“组织目标主线”与“下级BP汇总”

**问题**：个人 BP 自查模板以“个人 BP 目标”为主线即可闭环；但组织月报需要既体现“组织 BP 目标”的承诺兑现，又要纳入下级 BP 的贡献与偏差，避免只写组织抽象结论或只堆下级流水账，导致管理动作缺口。

**解决**：新增两份组织相关模板：组织 BP 自查模板以“组织目标”为底座，在每个组织目标中嵌入“下级BP贡献与偏差汇总”表，并额外提供“下级BP总览（管理动作入口）”页，要求每条偏差明确“组织动作 + 下级动作 + 最晚时间”；组织管理人评价模板要求同时参考“组织月报 + 管理人月报”，按组织目标逐条给出判断依据、管理动作复盘与下月可验收承诺，确保叙事一致且可执行。

### 4) 月报模板从“单模板”拆分为“三份独立报告模板”

**问题**：会议决议要求将原月报模板拆为三份独立报告（BP自查 / 个人总结 / 个人评价），且三份需要可单独阅读，同时又要保持灯色/证据引用等共用规范一致，避免口径漂移。

**解决**：将原 `cms-bp-monthly-report/references/report-template.md` 调整为“索引入口 + 共用灯色规范（色值表/三灯判断块）”；在同目录新增三份独立模板文件，分别承接会议定义的结构与读者定位，并在“个人总结报告”中以附录形式承接“BP自查报告”的目标级灯色结论，形成清晰的数据输入关系。

### 5) 黑灯不仅要整改/补证据，还要触发“是否调整 BP”的询问

**问题**：黑灯代表“已进入计划期但无有效证据”，仅要求补证据/整改容易遗漏更根本的判断：该节点是否需要调整（目标/KR/举措/时间范围/衡量标准等），导致月报闭环不足。

**解决**：在 `cms-bp-monthly-report/references/traffic-light-rules.md` 的“⚫ 黑灯-行动要求”中新增规则：命中黑灯时必须同步询问用户是否需要调整 BP，并在报告中记录用户选择与原因（不调整也需说明原因），与整改计划一起形成闭环。

### 6) 个人总结报告的“总体判断”应直接引用 BP自查的得分汇总

**问题**：报告二（个人总结）若在“1.1 本月总体判断”继续使用主观评级与四维标准，会与报告一（BP自查）的自评/自评表形成重复口径，且管理者更需要一眼看到“算出来的总分与目标权重影响”。

**解决**：将 `cms-bp-monthly-report/references/report-template-personal-summary.md` 的“1.1 本月总体判断”改为：直接展示来自报告一的本月最终加权得分、评级映射，以及按目标的“目标得分×目标权重”的明细表；叙事段落只解释分数变化原因，不再在报告二重算评分规则。

### 7) 管理者评分与管理者要求需要结构化字段，避免只写评语

**问题**：管理者对下级的评价如果只写“评语段落”，难以形成可追踪的下月要求，也难以与报告一（BP自查）的目标/权重/证据体系对齐，导致评分不可复盘、要求不可验收。

**解决**：在 `cms-bp-monthly-report/references/report-template-personal-summary.md` 新增第 5 章“管理者评分”和第 6 章“管理者要求”，用表格字段结构化输出：评分元信息、维度得分与权重、目标级（与目标权重挂钩）差异原因、下月硬性要求（含验收口径/截止时间/依赖/反证信号）、Stop Doing、能力与行为要求、以及双方跟踪机制。

### 8) 评分与评价过复杂时，回归“目标四维评分表”口径做极简化

**问题**：打分表/评价字段过多会降低管理者填写意愿，且容易出现“口径漂移”（不同人用不同权重/维度解释），最终失去横向可比性。

**解决**：参考 `cms-bp-monthly-report/references/goal-score-sheet.html` 的标准，把管理者评分统一为：每个目标按四维（目标达成度/推进质量/协同与影响力/风险与确定性）各打 1~5 分，采用固定权重（35/25/20/20）与护栏规则，换算 0~100 与五档评价；管理者要求收敛为 Top3 要求 + 1 条 Stop Doing + 支持与对齐节奏，保证最小闭环与可执行性。

### 9) “管理者要求”更适合强约束文本而非填表

**问题**：让管理者逐格填表容易造成输入负担，且输出读起来像数据表不够顺畅；但如果完全自由文本又会遗漏关键信息（验收口径/截止时间/依赖/反证信号等），导致要求不可执行。

**解决**：在 `cms-bp-monthly-report/references/report-template-personal-summary.md` 中将第 6 章改为“按固定结构输出一段文本”，强制包含 Top3 要求（每条含关联目标/动作/验收/截止时间/反证信号等）、Stop Doing（最多 1 条）以及管理者支持与对齐节奏，从而在“低负担”和“可落地”之间取得平衡。

### 10) 将第 5/6 章做成静态 HTML 填写页，降低填写成本

**问题**：即使模板已简化，管理者在 Markdown 里手填仍有摩擦（格式容易乱、汇总/换算麻烦、难导出复用）。

**解决**：新增 `cms-bp-monthly-report/references/manager-review-sheet.html` 静态页面：复用 `goal-score-sheet.html` 的 UI 与计算口径（四维 1~5 分、固定权重、护栏、0~100、五档评价、目标权重汇总）；同时提供“管理者要求文本”输入（可复制模板）、LocalStorage 自动保存、JSON 导出与一键清空。

### 11) 管理者打分时需要对照员工自评四维分数

**问题**：管理者在给每个目标打分时，如果看不到员工自评的四维分数，容易产生重复判断成本，也不利于对齐“差异在哪里”。

**解决**：在 `cms-bp-monthly-report/references/manager-review-sheet.html` 增加“导入员工自评 JSON”（使用 `goal-score-sheet.html` 的导出 JSON），并在每个目标的四维滑块提示中展示“员工自评”分数（例如“权重 0.35；员工自评 4”）；同时支持一键清空导入的自评分数。

**补充**：若暂无可导入 JSON，则页面内置一份“员工自评示例分数”作为预置（首次打开且未导入时自动展示），并提供“载入示例自评”按钮便于随时恢复示例。

### 1) 写入 Skill 需要“写前确认单/写后复核”，但写 Skill 原本只实现写接口

**问题**：`cms-bp-manager-write` 需要在执行 `updateTask` 等写操作前生成差异确认单，并在写后复核落地结果；但原实现只封装写接口，没有读取详情的能力。

**解决**：在写入侧的 `bp_client.py` 增加最小只读方法（`GetGoalDetail` / `GetKeyResultDetail` / `GetActionDetail`），仅用于写前/写后验证，不替代 `cms-bp-manager` 的查看/审计入口；CLI 的 `update-task` 以这些只读方法实现差异确认单与复核。

## 2026-04-09

### 13) 写操作必须以“版本链可回退”为兜底，并把回退流程前置到 Skill 协议里

**问题**：沟通中容易只强调“写前审计/二次确认”，但在实际落地时最常见的风险是“确认单仍会漏字段/误覆盖人员列表”。如果 Skill 文档没有把“版本历史查询/快照对比/回退”作为标准流程的一部分，使用者遇到误改时会陷入手工补救或用 `updateTask` 反向覆盖的高风险操作。

**解决**：在 `cms-bp-manager-write/SKILL.md` 中把 2.28～2.30（`getHistoryPage/getHistoryDetail/rollback`）提升为“基础操作场景”，要求回退前必须拉快照并生成差异确认单、回退后必须写后复核；同时强调优先用 2.30 原子回退，不用 `updateTask` 做复杂反向覆盖；补充 Long 雪花 ID 必须按字符串处理与参与人字段 `null`/`[]` 的覆盖语义，避免误清空角色。

### 2) 对齐脚本从“占位禁止编造”迁移到真实 OpenAPI

**问题**：历史占位脚本 `set_alignment.py` 仅提示“未接入禁止编造”，无法支持后端已上架的对齐接口。

**解决**：按 OpenAPI（二、2.26）接入 `POST /bp/task/v2/alignTask`，并用 `upwardTaskIdList=[]` 表达解除对齐。

### 3) 月报综述需要“主观评级 + 判断标准”，避免只给三灯结论

**问题**：月报综述段落（总体判断/关键进展/关注问题/下月判断）若只输出三灯或流水账，难以形成可复用的评审口径；也容易出现“主观判断缺少标准”导致的争议。

**解决**：在模板中为每个段落引入统一的四档主观评级（优秀/良好/合格/不足），并要求按维度给出可对照的判断标准与证据编号；同时将“下月判断”改为偏客观、可验证的口径，至少覆盖里程碑（含 Done 定义）、关键指标（含阈值与采集方式）、关键依赖与反证信号（触发调整的条件），从而把“预测”变成可复盘的判断。
