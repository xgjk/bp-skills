## 标准工作流：变更前审计 → 确认 → 写入 → 变更后复核

本文件用于把 `bp-manager` 的写入能力与 `bp-audit` 的审计规则，统一收敛为 `bp-manager-write` 的标准工作流。

### 适用范围

- 任何会改变 BP 数据的动作：目标/成果/举措的新增、修改、删除；对齐/承接关系的新增、修改、解除；延期提醒发送等。

### 核心原则（强制）

1. **先审计再写入**：所有写入前必须执行审计并输出风险点与建议。
2. **二次确认**：写入动作必须得到明确确认后才能执行。
3. **写入后复核**：写入完成后必须再次执行审计或关键检查项复核，确认变更达到预期且未引入新问题。
4. **不编造接口**：对齐/承接关系修改、编辑/删除等动作，如当前 Open API 未明确支持，禁止伪造接口；只能输出审计结论与人工操作建议，并保留“接口接入点”。

### 工作流分解

#### Step 0：定位对象

输入最小集（任一即可开始，但必须能唯一定位）：

- `groupId` / `taskId`
- 或者：`periodId` + 分组名称/任务名称（需先搜索定位）

输出要求：

- 明确写清楚“要变更的对象是谁”（目标/成果/举措/对齐关系）与其编号+名称（如已可得）

#### Step 1：变更前审计（Pre-Audit）

执行方式：

- 通过 `bp-manager-write/scripts/audit/audit_cli.py` 复用 `bp-audit` 的数据拉取与审计规则。

输出要求：

- 列出本次变更相关的关键风险点（结构/衡量标准/承接对齐/人员/周期/时间范围等）
- 每条问题必须“对象精确引用（编号+名称）+ 严重等级 + 原因 + 建议”

#### Step 2：形成变更方案（Plan）

输出要求：

- 写清楚“要改什么字段/关系、改前是什么、改后期望是什么”
- 说明影响面：是否影响上级对齐、下级承接、衡量标准可验收性

#### Step 3：二次确认（Confirm）

确认规则：

- 对 CLI 写操作：必须显式传入 `--confirm yes`
- 对对话式写操作：必须用户明确同意后才可执行

#### Step 4：执行写入（Write）

当前已接入写操作：

- 新增关键成果：`addKeyResult`
- 新增关键举措：`addAction`
- 发送延期提醒：`sendDelayReport`

未接入但必须保留场景：

- 目标/成果/举措编辑与删除
- 对齐/承接关系新增、修改、解除

这些未接入项必须走“接口接入点”（见下文），并在输出中明确提示需前端人工操作。

#### Step 5：变更后复核（Post-Check）

执行方式：

- 复用审计入口再跑一次（或至少跑与本次变更相关的板块）

输出要求：

- 说明“变更是否达成预期、是否引入新问题、还有哪些缺口需要补”

### 对齐/承接关系修改：接口接入点（占位）

当前仓库未落地对齐/承接关系修改的明确 Open API/脚本实现，因此在 `bp-manager-write/scripts/write/` 中保留占位脚本作为接入点：

- `set_alignment.py`：用于未来接入“设置/修改/解除对齐承接关系”的真实接口调用

占位脚本当前行为必须是：

- 明确报错“接口未接入”
- 输出需要你们提供的接口契约信息清单（路径、方法、入参、鉴权模式、幂等与错误码）
