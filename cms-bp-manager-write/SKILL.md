# BP Manager Write（纯写入）

> BP 写入助手 — 新增关键成果/关键举措、发送延期提醒（受控二次确认）

---

## 角色定位

本技能仅承载 BP 的**写入更新**能力，所有查看/搜索/审计能力已归入 `cms-bp-manager`。

**当前版本**: 2.0.0  
**接口版本**: v1（BP Open API，`/open-api/bp/*`）

---

## 核心场景

### 场景一：新增关键成果

**用户意图**：为某个目标添加新的关键成果

**触发词**：
- "为目标X新增关键成果"
- "添加一个关键成果"

**执行流程**：
1. 解析用户意图，识别目标ID
2. 收集必要信息：
   - 关键成果名称（必填）
   - 计划起止时间（可选）
   - 承接人（可选）
   - 衡量标准（可选）
3. **二次确认**
4. 调用 `addKeyResult` 接口
5. 确认创建结果

**示例**：
```
用户：为目标 A4-1 新增一个关键成果：完成全栈交付项目模式探索的报告
助手：确认要新增关键成果【完成全栈交付项目模式探索的报告】到目标 A4-1？(yes/no)
用户：yes
助手：已成功创建关键成果【完成全栈交付项目模式探索的报告】
关键成果ID: 2014631829004376001
```

### 场景二：新增关键举措

**用户意图**：为某个关键成果添加关键举措

**触发词**：
- "为成果X新增关键举措"
- "添加一个关键举措"

**执行流程**：
1. 解析用户意图，识别关键成果ID
2. 收集必要信息：
   - 关键举措名称（必填）
   - 计划起止时间（可选）
   - 承接人（可选）
   - 衡量标准（可选）
3. **二次确认**
4. 调用 `addAction` 接口
5. 确认创建结果

**示例**：
```
用户：为成果 1.1 新增关键举措：每周拜访5家客户
助手：确认要新增关键举措【每周拜访5家客户】到成果 1.1？(yes/no)
用户：yes
助手：已成功创建关键举措【每周拜访5家客户】
关键举措ID: 2014631829004376002
```

---

### 场景三：延期提醒

**用户意图**：向指定员工发送延期提醒

**触发词**：
- "发送延期提醒给张三"
- "提醒李四他的 BP 延期了"

**执行流程**：
1. 识别目标员工
2. 收集延期任务信息：
   - 任务名称
   - 计划结束日期
   - 当前状态
3. 生成提醒内容
4. **二次确认**
5. 调用 `sendDelayReport` 接口
6. 确认发送结果

**示例**：
```
用户：发送延期提醒给张三，他的关键成果延期了
助手：请提供以下信息：
- 延期的任务名称
- 计划结束日期
- 您想提醒的内容

用户：关键成果【客户拜访量达到50家】，计划结束日期是2026-02-28
助手：确认发送延期提醒给张三？(yes/no)
用户：yes
助手：已成功发送延期提醒给张三
```

## 写入安全约束（强制）

1. **先审计再写入**：所有写入前必须执行审计并输出风险点与建议（审计使用 `cms-bp-manager`）
2. **二次确认**：写操作必须得到明确确认后才能执行（CLI 层面：`--confirm yes`）
3. **写入后复核**：写入完成后必须再次检查，确认变更达到预期且未引入新问题
4. **不编造接口**：对齐/承接关系修改、编辑/删除等动作，如当前 Open API 未明确支持，禁止伪造

---

## 建议工作流

```
Step 0: 定位对象（groupId/taskId 或搜索定位）
Step 1: 变更前审计 → cms-bp-manager check-bp
Step 2: 形成变更方案（改什么、改前什么、改后什么）
Step 3: 二次确认
Step 4: 执行写入 → cms-bp-manager-write
Step 5: 变更后复核 → cms-bp-manager check-bp
```

工作流详情见：`./references/workflow/README.md`

## API 接口清单（写入）

| 接口 | 方法 | 用途 |
|-----|------|------|
| `POST /bp/task/v2/addKeyResult` | 新增关键成果 | |
| `POST /bp/task/v2/addAction` | 新增关键举措 | |
| `POST /bp/delayReport/send` | 发送延期提醒汇报 | |

---

## 不能做什么（当前阶段）

- 对齐/承接关系"修改设置"如无明确 Open API 支持，禁止伪造接口；只能给出审计结论与人工操作建议。
- 目标/成果/举措的"编辑/删除"如无明确 Open API 支持，禁止伪造接口；只能给出变更方案与人工操作建议。

---

## 宪章

1. 写入更新必须强制联动审计（使用 `cms-bp-manager`），防止"改了但不合规/不承接"
2. 未接入接口必须明确提示并给出人工操作建议，禁止编造
3. 不在输出中泄露 `appKey`、`access-token`
