# BP Manager Write（纯写入）

> BP 写入助手 — 新增关键成果/关键举措、发送延期提醒、承接/对齐、通用修改、版本回退（受控二次确认 + 可回滚）

---

## 角色定位

本技能仅承载 BP 的**写入更新**能力，所有查看/搜索/审计能力已归入 `cms-bp-manager`。

**当前版本**: 2.1.1  
**接口版本**: v1（BP Open API，`/open-api/bp/*`）

---

## 核心场景

### 场景一：新增关键成果

**用户意图**：为某个目标添加新的关键成果

**触发词**：
- "为目标X新增关键成果"
- "添加一个关键成果"

**执行流程**：
1. 解析用户意图，识别目标ID
2. 收集必要信息：
   - 关键成果名称（必填）
   - 计划起止时间（可选）
   - 承接人（可选）
   - 衡量标准（可选）
3. **二次确认**
4. 调用 `addKeyResult` 接口
5. 确认创建结果

**示例**：
```
用户：为目标 A4-1 新增一个关键成果：完成全栈交付项目模式探索的报告
助手：确认要新增关键成果【完成全栈交付项目模式探索的报告】到目标 A4-1？(yes/no)
用户：yes
助手：已成功创建关键成果【完成全栈交付项目模式探索的报告】
关键成果ID: 2014631829004376001
```

### 场景三：延期提醒

**用户意图**：向指定员工发送延期提醒

**触发词**：
- "发送延期提醒给张三"
- "提醒李四他的 BP 延期了"

**执行流程**：
1. 识别目标员工
2. 收集延期任务信息：
   - 任务名称
   - 计划结束日期
   - 当前状态
3. 生成提醒内容
4. **二次确认**
5. 调用 `sendDelayReport` 接口
6. 确认发送结果

**示例**：
```
用户：发送延期提醒给张三，他的关键成果延期了
助手：请提供以下信息：
- 延期的任务名称
- 计划结束日期
- 您想提醒的内容

用户：关键成果【客户拜访量达到50家】，计划结束日期是2026-02-28
助手：确认发送延期提醒给张三？(yes/no)
用户：yes
助手：已成功发送延期提醒给张三
```

### 场景四：修改任务（通用更新）

**用户意图**：修改某个 Goal/KR/Action 的内容或参与人（名称、时间、权重/衡量标准、责任人/协办/抄送/监督/观察、责任部门）

**触发词**：
- "把目标 A4-1 截止时间改到 2026-06-30"
- "把 KR A4-1.1 的承接人改为张三"
- "把举措 A4-1.1.1 的衡量标准补充一下"
- "把这个目标的权重从 40 调整到 30"

**执行流程（强制）**：
1. 定位任务 `taskId`（必要时让用户提供；或引导使用 `cms-bp-manager` 先查到 ID）
2. 拉取变更前现状（用于差异确认与写后复核）：
   - 目标：`GET /bp/goal/{goalId}/detail`
   - 成果：`GET /bp/keyResult/{keyResultId}/detail`
   - 举措：`GET /bp/action/{actionId}/detail`
3. 生成“差异确认单”（变更前/变更后对比），并明确：
   - 改动字段清单（只列出本次要改的字段）
   - 影响范围（是否会清空角色/覆盖人员列表）
   - 风险提示（敏感字段/时间范围/对齐关系）
4. **二次确认**（必须得到明确 yes 才能执行）
5. 调用 `POST /bp/task/v2/updateTask`
6. 写后复核：再次拉取详情，确认变更已生效且未引入意外变更
7. 如发现误改，立即走“版本回退”（见场景六）

**关键规则（必须遵守）**：
- 本接口为“通用更新”，不等价于“编辑页面所有字段”；**只能提交必要字段**，避免误覆盖。
- 参与人类字段的空值语义：
  - 不传（`null`）：保持原样
  - 传空数组（`[]`）：清空该角色（高风险，必须在确认单中显式提示）
  - 传具体列表：全量覆盖

**示例（对话）**：
```
用户：把 KR 2014631829004374018 的计划结束时间改到 2026-03-15，并把协办人加上李四
助手：我将做如下修改（请确认）：
- taskId=2014631829004374018
- planEndDate：2026-02-28 → 2026-03-15
- collaboratorIds：原=[...] → 新=[..., 李四]
是否执行？(yes/no)
用户：yes
助手：已更新成功，并已复核详情一致。如需回退可提供 snapshotId。
```

### 场景五：查看版本历史与快照详情（用于审计/差异对比）

**用户意图**：查看某任务的历史修改记录，或获取某个版本的全量快照用于对比

**触发词**：
- "看看这个任务之前改过什么"
- "把这个任务回退前先把历史版本列出来"
- "拉一下 v3 的快照看看当时内容"

**执行流程**：
1. 定位 `taskId`
2. 调用 `GET /bp/task/v2/getHistoryPage?taskId=...` 获取版本流水
3. 用户指定某条记录后，调用 `GET /bp/task/v2/getHistoryDetail?id={snapshotId}` 获取全量快照
4. 生成“快照 vs 当前”的差异摘要（用于后续回退确认）

**输出约束**：
- 快照中的 Long 型 ID 在 JS/前端等环境必须按**字符串**处理，禁止 `Number()` / `parseInt` 直接转数值。

---

### 场景六：版本回退（后悔药）

**用户意图**：把任务恢复到某个历史快照版本（误改修复、恢复正确字段）

**触发词**：
- "回退到上一个版本"
- "撤销刚才的修改"
- "把它恢复成昨天的状态"

**执行流程（强制）**：
1. 定位 `taskId`
2. 调用 `getHistoryPage` 列出版本（2.28）
3. 选定目标 `snapshotId` 后，先调用 `getHistoryDetail`（2.29）拉快照并生成差异确认单
4. **二次确认**（必须明确 yes；并在确认单里写清楚“将回退到哪个版本、由谁在何时生成”）
5. 调用 `POST /bp/task/v2/rollback`（2.30）
6. 写后复核：再次拉取详情，确认已恢复到目标状态

**重要说明**：
- “回退”是原子操作，优先使用 2.30；不要依赖 `updateTask` 做复杂反向覆盖。
- “撤销新建/删除节点”不等价于回退：若没有删除 API，上层只能做到回退到历史时刻，无法保证实现“删除新建节点”的语义。

## 写入安全约束（强制）

1. **先审计再写入**：所有写入前必须执行审计并输出风险点与建议（审计使用 `cms-bp-manager`）
2. **二次确认**：写操作必须得到明确确认后才能执行（CLI 层面：`--confirm yes`）
3. **写入后复核**：写入完成后必须再次检查，确认变更达到预期且未引入新问题
4. **不编造接口**：对齐/承接关系修改、编辑/删除等动作，如当前 Open API 未明确支持，禁止伪造
5. **Long ID 字符串化**：所有 Long 型雪花 ID 在 JS 等环境必须按字符串处理，禁止当作 number 运算或比较
6. **中文 URL 编码**：凡是 `name` 模糊搜索类参数（若写入流程中需要引导用户查询），必须 UTF-8 URL 编码，避免请求失败

## 关于“权限、版本与回退”的统一答复（现状说明）

> 适用范围：本 Skill 的所有写操作（新增关键成果/关键举措、延期提醒，以及承接/对齐/更新/回退等写接口）。

### 1) 关于“权限判断支持缺失”的说明

后端已闭环：当前写接口并非“裸奔”。底层服务会基于 Token 识别当前操作人，并对每一次写操作执行强校验：仅当当前人为该任务的**负责人**、**主责部门负责人**或**分组管理员**时，才允许修改/对齐/回退；无权限的调用无法通过接口修改或回退任何数据。

未来扩展：你们提到的“字段级白名单”（例如 AI 只能改时间不能改负责人）属于更细粒度的微观安全策略，建议在 Skill 脚本侧实现**调用前拦截**与二次确认单的差异展示（见第 4 点）。

### 2) 关于“版本查询与记录缺失”的说明

原生支持，无需本地存储：Skill 侧不需要维护 `.cms-log/` 之类的本地版本文件。后端已支持自动化快照机制：当调用写接口（例如 `addGoal`、`updateTask`、`alignTask` 等）成功后，会自动生成包含操作人、时间、全量字段的 JSON 快照并写入历史库。

接口已上架：可直接通过 OpenAPI 获取历史版本链路：

- `GET /bp/task/v2/getHistoryPage`：查询任务版本历史列表（接口定义二、2.28）
- `GET /bp/task/v2/getHistoryDetail`：获取指定快照的全量详情（接口定义二、2.29）

### 3) 关于“版本回退能力”的实现路径

原生原子回退：不建议依赖 `updateTask` 去做复杂的“反向覆盖”。后端已暴露原子回退接口：

- `POST /bp/task/v2/rollback`：传入 `snapshotId` 一键回退（接口定义二、2.30）

闭环建议：对于“新增/创建”类操作，如果业务要支持“撤销新建”，应在 Skill 侧根据操作类型判断，配套调用后端将要补充的“删除任务”接口；在删除能力未上架前，Skill 只能做到“版本回退”而不能做“撤销创建”。

### 4) 关于“修改内容对比”与“二次确认单”

数据已就绪：由于快照详情返回的是结构化的全量 JSON（见 `TaskVersionSnapshotVO.snapshotData`），对比逻辑建议由 Skill 侧实现；展示层（把员工/部门 ID 翻译成人名，把状态码翻译成人话）与前端诉求高度相关，不应强耦合到后端。

分工建议：Skill 在执行写操作前，至少做一次“变更前现状拉取”，再与目标变更/历史快照进行 Key-Value diff，生成结构化确认单据后再执行写入。推荐读取路径：

- 变更前现状：按对象类型调用详情接口（2.5/2.6/2.7）或按分组通读 `getGroupMarkdown`（2.13），并保留调用链中的 `groupId`（注意：详情接口响应体不带 `groupId`）
- 历史版本：`getHistoryDetail`（2.29）拉取 `snapshotData`
- 回退执行：`rollback`（2.30）

总结：后端已提供完整的“后悔药”机制（记录 + 查询 + 回退）。Skill 侧应专注处理：① 敏感字段拦截；② 变更前后差异可视化与二次确认。

## 建议工作流

```
Step 0: 定位对象（groupId/taskId 或搜索定位）
Step 1: 变更前审计 → cms-bp-manager check-bp
Step 2: 形成变更方案（改什么、改前什么、改后什么）
Step 3: 二次确认
Step 4: 执行写入 → cms-bp-manager-write
Step 5: 变更后复核 → cms-bp-manager check-bp
```

工作流详情见：`./references/workflow/README.md`

## 不能做什么（当前阶段）

- 对齐/承接关系"修改设置"如无明确 Open API 支持，禁止伪造接口；只能给出审计结论与人工操作建议。
- 目标/成果/举措的"编辑/删除"如无明确 Open API 支持，禁止伪造接口；只能给出变更方案与人工操作建议。
- 对“新建节点的撤销/删除”当前无明确 Open API 支持时，禁止伪造删除；只能通过 **2.30 版本回退** 实现“回到某个历史时刻”，无法等价替代“撤销创建”。

## 宪章

1. 写入更新必须强制联动审计（使用 `cms-bp-manager`），防止"改了但不合规/不承接"
2. 未接入接口必须明确提示并给出人工操作建议，禁止编造
3. 不在输出中泄露 `appKey`、`access-token`
